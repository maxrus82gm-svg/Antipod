# 07_Правила_Git_и_Review

## 1. Назначение документа

Этот документ описывает правила безопасной работы с Git, GitHub, commit review, diff, status и историей изменений в проекте Antipod.

Документ читается только тогда, когда задача связана с Git, GitHub, diff, status, review, историей изменений или проверкой commit.

Перед обычными задачами этот документ читать не нужно.

---

## 2. Когда читать этот документ

Агент обязан прочитать этот документ, если задача связана с:

- Git;
- GitHub;
- status;
- diff;
- commit;
- push;
- pull;
- branch;
- review;
- историей изменений;
- проверкой репозитория;
- проверкой последнего commit;
- проверкой, был ли push;
- сравнением локального состояния и GitHub.

Если задача не связана с Git или review, этот документ читать не нужно.

## 4. Главное правило Git

Git контролирует только пользователь.

Агент не имеет права выполнять действия, которые меняют состояние Git-репозитория.

Агенту запрещено выполнять:

```bash
git add
git commit
git push
git pull
git merge
git rebase
git reset
git clean
git checkout
git switch
git branch
git tag
```

Агент не создаёт commit.  
Агент не делает push.  
Агент не делает pull.  
Агент не создаёт ветки.  
Агент не переключает ветки.  
Агент не делает merge.  
Агент не делает rebase.  
Агент не откатывает изменения.  
Агент не чистит рабочую папку.  
Агент не управляет историей Git.

Этими процессами управляет только пользователь.

---

## 5. Что агенту разрешено

Агент может только читать состояние Git, если это нужно для задачи.

Разрешены только безопасные команды просмотра:

```bash
git status
git branch --show-current
git remote -v
git log --oneline -5
git diff --stat
git diff
```

Эти команды не должны менять файлы, ветки, историю или состояние репозитория.

Если агент использовал Git-команды, он обязан указать это в отчёте.

---

## 6. Запрещённые Git-действия

Агенту запрещено:

- индексировать файлы через `git add`;
- создавать commit;
- отправлять изменения через push;
- получать изменения через pull;
- делать merge;
- делать rebase;
- делать reset;
- делать clean;
- делать checkout;
- делать switch;
- создавать ветки;
- переключать ветки;
- удалять ветки;
- создавать tag;
- менять историю Git.

Даже если задача связана с Git, агент выполняет только анализ и отчёт.

---

## 7. Правило перед изменением файлов

Перед изменением файлов агент может проверить состояние проекта безопасной командой:

```bash
git status
```

Если есть незакоммиченные изменения, агент обязан указать это в отчёте.

Агент не должен перезаписывать пользовательские изменения.

Если есть риск потери данных, агент обязан остановиться и написать об этом в отчёте.

---

## 8. Правило commit и push

Commit и push выполняет только пользователь.

Агент не должен предлагать выполнить commit или push как своё действие.

Правильная формулировка в отчёте:

```text
Изменения готовы к проверке пользователем.
Commit/push выполняет пользователь самостоятельно.
```

Неправильная формулировка:

```text
Сейчас сделаю commit и push.
```

---

## 9. Правило Review

Если задача является review после работы агента или Codex, ответ должен начинаться со статуса commit.

Формат:

```text
СТАТУС КОММИТА: "<commit message>"
Repository: maxrus82gm-svg/Antipod
Branch/ref: main
Commit: <short sha>
Проверка: изменения найдены / push не обнаружен / требуется уточнение
```

Если commit message доступен, нужно указать именно его.

Если commit message короткий или неинформативный, например `+`, `01`, `02`, агент всё равно указывает его как есть.

Если commit message не подтверждён, агент честно пишет:

```text
Commit message: не подтверждён
Причина: ...
```

---

## 10. Статус чтения GitHub-репозитория

Каждый раз, когда агент читает данные из GitHub-репозитория, он обязан явно указать статус чтения репозитория.

Минимальный формат:

```text
Repository:
Branch/ref:
Commit SHA:
Commit message:
Files read:
```

Если файл читается через default branch или `main`, нужно подтвердить текущий commit этой ветки и указать его в ответе.

Если пользователь дал конкретный hash, агент должен написать, что чтение выполнено именно по этому hash.

Если commit не удалось подтвердить, нужно указать:

```text
Commit SHA: не подтверждён
Причина: ...
```

Если агент читает несколько файлов, он должен перечислить все ключевые прочитанные файлы.

---

## 11. Правило локальных изменений

Агент не должен утверждать, что видит локальные изменения пользователя, если он их реально не проверил.

Если агент проверяет только GitHub, он должен писать:

```text
Проверены данные на GitHub.
Локальные изменения не проверялись.
```

Если агент проверяет локальный репозиторий, он должен указать:

```text
Проверен локальный путь:
M:\GitHub\Antipod\Progekt_antipod
```

---

## 12. Формат отчёта для Git-задачи

Если задача затрагивала Git, агент должен указать:

```text
Branch:
- текущая ветка

Status:
- есть ли незакоммиченные изменения

Changed files:
- какие файлы изменены

Diff summary:
- кратко что изменилось

Commit:
- не создавался агентом

Push:
- не выполнялся агентом

GitHub check:
- проверялся или нет
```

---

## 13. Главное правило

Git используется только для контроля и анализа изменений.

Главный порядок работы агента:

```text
посмотреть status
понять изменения
сделать маленькую правку, если это часть задачи
проверить diff
написать отчёт
остановиться
ждать действий пользователя
```

Commit, push, pull, branch, merge и другие Git-действия выполняет только пользователь.
